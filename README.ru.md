# TcpWireProtocol

[English](README.md) | **Русский**

Небольшая библиотека для C#, которая снимает одну надоедливую задачу при работе с TCP:
как слать и принимать целые сообщения, а не поток байтов. Заодно она шифрует их.

## Зачем это нужно

TCP отдаёт вам поток байтов, а не сообщения. Вы отправили один пакет на 200 байт, а на
другом конце `Receive` вернёт сначала 150 байт, потом остальные 50. Или, наоборот, склеит
два ваших пакета в один буфер. Границ между сообщениями в TCP просто нет, их приходится
размечать самому.

Обычно делают так: перед каждым сообщением пишут его длину, а на приёме сначала читают эту
длину и ждут, пока из сокета наберётся столько байтов. Код скучный, обязательный и на нём
легко ошибиться: недочитал, склеил, поехал на границе кадра, и весь поток рассыпался.
TcpWireProtocol берёт эту рутину на себя и вдобавок шифрует каждое сообщение через AES-GCM,
так что по проводу идут не просто размеченные, а ещё и защищённые кадры.

Чего библиотека **не** делает, так это не открывает сокеты и вообще не трогает сеть. Транспорт
остаётся ваш: `Socket`, `NetworkStream`, канал, заглушка в тесте. Библиотека сидит тонкой
прослойкой между этим транспортом и вашей логикой: на выходе превращает объект-сообщение в
готовый к отправке кадр, на входе собирает кадр обратно и расшифровывает.

## Три вида сообщений

Модель - клиент и сервер. Сообщения бывают трёх видов:

- **`Request`** - запрос от клиента к серверу. Несёт номер (`CmdId`), по которому потом можно
  сопоставить ответ с запросом.
- **`Response`** - ответ сервера на конкретный запрос. Несёт тот же `CmdId`.
- **`Event`** - сообщение, которое сервер шлёт сам, без запроса (например, «пришёл новый
  пользователь»).

Куда именно направить сообщение, решают два числа: `service` выбирает подсистему (скажем,
авторизацию), а `command` - конкретную операцию внутри неё (войти, выйти, зарегистрироваться).

## С чего начать

Логика всегда одна и та же. На отправке: собрали сообщение, попросили кодек упаковать его в
кадр, отправили этот кадр своим транспортом.

```csharp
using TcpWireProtocol.Packets;
using TcpWireProtocol.Security;

// Кодек создаётся по одному на каждую сторону соединения. Внутри он считает кадры, поэтому
// один экземпляр = одно соединение, переиспользовать между соединениями нельзя.
// key == null означает "без реальной защиты" (см. раздел про шифрование ниже).
IWireCodec codec = WireCodec.Create(WireDirection.ClientToServer, key);   // на клиенте
// на сервере:      WireCodec.Create(WireDirection.ServerToClient, key)

var request = new Request(service: 1, command: 1, payload);
transport.Send(codec.Encode(request));      // один зашифрованный кадр с длиной впереди
```

На приёме сложнее ровно настолько, насколько сложен сам TCP: у вас на руках есть только то,
что уже успело прийти из сокета, и целого кадра там может пока не быть. Поэтому приём - это
цикл. Вы отдаёте кодеку накопленный буфер, а он отвечает одно из трёх:

- **`NeedMoreData`** - кадр ещё не пришёл целиком, дочитайте из сокета и попробуйте снова;
- **`Ok`** - кадр разобран, вот расшифрованные байты, и вот сколько байтов буфера он занял;
- **`Corrupt`** - подделка, повтор или неверный ключ; такое соединение надо закрывать.

```csharp
// buffer[offset..offset+count] - то, что уже прочитано из транспорта.
while (true)
{
    var status = codec.TryDecode(buffer, offset, count, out byte[]? raw, out int consumed);

    if (status == WireDecodeStatus.NeedMoreData)
        break;                          // ждём ещё байтов из сокета

    if (status == WireDecodeStatus.Corrupt)
    {
        transport.Close();              // поток уже не пересобрать, рвём соединение
        break;
    }

    // Ok: raw - расшифрованное сообщение, кадр занял consumed байт.
    offset += consumed;
    count  -= consumed;

    Handle(raw);
}
```

Дальше расшифрованные байты превращаются обратно в типизированное сообщение через `TryParse`.
На сервере входящее читается как `Request`, а ответ удобно собрать прямо из него: `CreateResponse`
подставит правильный `CmdId`:

```csharp
if (Request.TryParse(raw, out Request request))
{
    var response = request.CreateResponse(replyPayload);
    transport.Send(codec.Encode(response));
}
```

На клиенте всё входящее читается как `Response`. Тонкость: события (`Event`) приходят по тому
же каналу, что и ответы, и намеренно устроены так, чтобы читаться как обычный `Response`.
Отличить их можно по зарезервированному `CmdId == 0` - для этого есть `IsEvent` и `TryReadEvent`,
который перечитывает тот же буфер как событие (без копирования):

```csharp
if (Response.TryParse(raw, out Response response))
{
    if (response.TryReadEvent(out Event evt))
        HandleEvent(evt);           // сервер прислал событие сам
    else
        HandleResponse(response);   // это ответ на наш запрос, ищем его по CmdId
}
```

Так что отдельного байта "тип сообщения" в протоколе нет и не нужно: направление соединения
уже отделяет запрос от ответа, а `CmdId == 0` - событие от ответа.

Если целый кадр у вас уже лежит в буфере (например, в тесте), цикл не нужен. Для этого есть
короткая перегрузка: она принимает готовый кадр целиком и возвращает `true`/`false` вместо
трёх состояний.

```csharp
if (codec.TryDecode(frame, out byte[]? raw)) { /* raw - расшифрованное сообщение */ }
```

## Шифрование

Шифрование живёт в кодеке и никак не задевает сами сообщения: `Request`, `Response`, `Event`
и их разбор о нём не знают. Под капотом - AES-GCM: он в один проход и шифрует, и проверяет
целостность, да ещё и с аппаратным ускорением. Ключ - 32 байта (AES-256), других вариантов нет,
чтобы не разводить настроек. Если кадр подделали, повторили или пришёл он под чужим ключом,
расшифровка не проходит, и вы получаете `Corrupt`. Тихо "переварить" испорченные данные
протокол не даст.

Режима "без шифрования" тоже нет, и это сознательно, чтобы формат кадра и путь кода были
одни на все случаи. Если ключ не передать, кодек берёт общеизвестный нулевой ключ
(`WireCodec.ZeroKey`): формально всё шифруется, но такой ключ знает кто угодно, кто видел
исходники, так что защиты в этом режиме нет, это просто "открытый протокол". Настоящую защиту
даёт настоящий ключ: заранее согласованный или полученный вашим кодом из обмена ключами
(например, Диффи-Хеллмана).

Самое интересное - как устроен nonce (одноразовое значение для AES-GCM). Обычно его гоняют по
проводу вместе с каждым сообщением. Здесь его не передают вообще: обе стороны считают его сами
из направления и счётчика кадров, который стартует с нуля на каждом соединении. Это работает
именно потому, что снизу TCP: он доставляет кадры по порядку и без потерь, поэтому счётчик
отправителя и ожидаемый счётчик получателя идут след в след. А приятный побочный эффект такой:
если кто-то повторит, переставит местами или подкинет старый кадр, у получателя он попадёт на
"не тот" счётчик, nonce не совпадёт, и проверка провалится. То есть защита от повторов
получается бесплатно, сама собой.

Одна деталь, о которой честно стоит знать: длина сообщения идёт по проводу открыто (её
шифровать нельзя, по ней как раз и определяют границы кадра). Подделать её незаметно нельзя,
её значение подмешано в проверку целостности тела. Но пассивный наблюдатель видит, где кончается
одно сообщение и начинается другое, и может прикинуть их размеры. Это плата за скорость - один
проход шифрования вместо двух. Если размеры нужно прятать, дополняйте сообщения до одинаковой
длины или берите двухпроходную схему.

Несколько правил, которые важно соблюдать:

- один кодек на одну сторону соединения, он с состоянием и не потокобезопасен;
- счётчики живут только в памяти и обнуляются на новом соединении, поэтому если ключ сессионный,
  меняйте его на каждую сессию;
- получили `Corrupt` - закрывайте соединение, пересобрать поток после этого уже не получится.

### Ограничение на размер

У кодека есть необязательный параметр `maxMessageLength` (по умолчанию 1 МБ). Если во входящем
кадре длина больше лимита, кодек вернёт `Corrupt` ещё до того, как выделит хоть байт памяти.
Это защита от ситуации, когда кто-то присылает мусорную гигантскую длину, чтобы посадить вам
память. Фабрика `WireCodec.Create` использует значение по умолчанию; нужно другое - создавайте
кодек напрямую:

```csharp
var codec = new CounterAesGcmWireCodec(key, WireDirection.ClientToServer, maxMessageLength: 64 * 1024);
```

## Установка

```
dotnet add package TcpWireProtocol
```

Собрано под `netstandard2.1`, `net8.0` и `net10.0`. Сборка `netstandard2.1` покрывает
.NET Core 3.0+, .NET 5+, Mono 6.4+, Xamarin и Unity 2021.2+. Классический .NET Framework не
поддерживается, в нём нет `AesGcm`. Сторонних зависимостей нет ни на одной платформе.

## Формат кадра

Если нужно реализовать совместимую сторону на другом языке - вот раскладка. Все многобайтовые
поля пишутся в little-endian. Ниже показана начинка *сообщения* (то, что читает `TryParse`);
на проводе кодек оборачивает её в кадр `длина(4) | тег(16) | зашифрованное тело`, а nonce, как
сказано выше, не передаётся, а выводится из направления и счётчика.

**`Request`**

| cmd id | length | service | command | payload |
| --- | --- | --- | --- | --- |
| int (4) | int (4) | short (2) | short (2) | byte[] (*length*) |

**`Response`**

| cmd id | length | payload |
| --- | --- | --- |
| int (4) | int (4) | byte[] (*length*) |

**`Event`**

| cmd id | length | service | command | payload |
| --- | --- | --- | --- | --- |
| int (4) | int (4) | short (2) | short (2) | byte[] (*length*) |

У `Event` поле `cmd id` всегда `0`, а `length` включает в себя размер `service` и `command`.
Именно поэтому событие и читается как обычный `Response`: первые поля `cmd id` и `length`
совпадают, а `service` с `command` оказываются внутри того, что `Response` считает началом тела.
